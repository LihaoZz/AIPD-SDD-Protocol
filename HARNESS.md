# Harness

## Purpose

The harness layer turns one `MB` into a machine-runnable execution unit.

Use this layer to:

- read one active mission block and its machine spec
- run one bounded Builder loop through Codex CLI
- enforce file-scope boundaries
- run machine-checkable verification
- persist runtime state and cross-session memory
- stop, retry, or route to recovery using explicit rules

The harness does not replace protocol truth.

Authority order remains:

1. active `FB`
2. active `MB`
3. active `MB` machine spec
4. quality and runtime artifacts

## Canonical Naming

Use one canonical identifier set across markdown, JSON, runtime state, logs, and memory.

- `fb_id`
- `mb_id`
- `parent_fb_id`
- `attempt_id`
- `last_attempt_id`
- `last_verification_report_path`
- `last_verification_digest`

Do not create alternate keys for the same object such as `fb`, `parent`, `verification_file`, or similar aliases.

## Mission Sidecar Rule

Every runnable mission block must have two files with the same base id:

- `<PROJECT_ROOT>/missions/<mb_id>.md`
- `<PROJECT_ROOT>/missions/<mb_id>.machine.json`

The markdown file is the human-readable source of truth.

The machine file is the executable sidecar used by validation, preflight, runner, guard, verifier, and state writing.

## Runtime Layout

Runtime artifacts live under the external `PROJECT_ROOT`.

Recommended layout:

```text
<PROJECT_ROOT>/
├── missions/
│   ├── fb1-mb1.md
│   └── fb1-mb1.machine.json
├── runtime/
│   ├── attempts/
│   │   └── fb1-mb1/
│   │       └── attempt-001/
│   ├── state/
│   │   └── fb1-mb1.state.json
│   ├── memory/
│   │   ├── project_memory.json
│   │   └── failure_log.json
│   └── preflight/
│       ├── session_preflight.json
│       └── mb_preflight_fb1-mb1.json
└── SESSION_STATE.md
```

## Verification Report vs Digest

These are different artifacts and must not share the same meaning.

- `verification_report.json` is the full evidence file written to one attempt directory
- `last_verification_digest` is a compact summary derived from the latest report for retry prompt injection

`last_verification_digest` is not an independent source-of-truth file.

It is generated by `mb_runner` from `last_verification_report_path` before the next retry prompt is assembled.

## Retry Feedback Rule

If one attempt fails and retry is still allowed, `mb_runner` must inject all of the following into the next prompt:

1. `last_verification_digest`
2. `last_failure_reason`
3. `retry_count`

Do not retry with a generic "try again" prompt.

## Structural Route Rule

If the Builder explicitly reports a structural blocker such as:

- `spec_gap`
- `state_drift`
- `environment_issue`
- `review_context_gap`
- blocked contract or missing-input conflicts that prevent a compliant run

then `mb_runner` must route the attempt to recovery before verification retry.

Do not collapse an explicit structural block into a generic verification retry.

## Dual Preflight Rule

Use one preflight script with two levels:

- `session`: checks whether the project or conversation can start safely
- `mb`: checks whether one specific mission block can be executed by the harness now

`mb_runner` must always run `mb` preflight before invoking Codex CLI.

## Runtime State Rule

Machine state lives in:

- `<PROJECT_ROOT>/runtime/state/<mb_id>.state.json`

Suggested status values:

- `ready`
- `running`
- `verifying`
- `passed`
- `failed`
- `blocked`
- `routed_to_recovery`

`SESSION_STATE.md` remains the human-readable bridge summary.

The machine runtime state is the execution source of truth.

For runnable `MB`s, `SESSION_STATE.md` is runtime-managed.

Do not list `SESSION_STATE.md` under `required_artifact_updates` in a runnable mission block.

The Builder may read `SESSION_STATE.md`, but session-state syncing belongs to the harness runtime.

## Non-Git Workspace Rule

The harness may run against a plain directory instead of a Git repository.

In that case:

- `scope_guard.py` remains the file-boundary authority
- the Builder should not use `git status` or `git diff` just to report changed files
- changed-file reporting should describe direct edits instead of depending on repository diff commands

## Timestamp Rule

All harness runtime timestamps must be recorded in the local timezone active at execution time.

Do not default runtime JSON timestamps to UTC.

Use ISO 8601 with offset for the actual timestamp string, and also store the local timezone name.

Recommended fields:

- `updated_at`
- `generated_at`
- `timezone_name`
